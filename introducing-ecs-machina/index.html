Introducing ECS Machina

More than a year and a half ago, I started to build my very own HTML game engine. In the meantime, I changed jobs, had one more kid (this is getting out of hand! Now there are two of them!), and so, my engine took a backseat. I had a hard time to stay motivated on this project, and started to fear that I might not be able to finish it before losing all steam. I then decided to split it into two different projects, since it's easier to finish two smaller tasks than a big one.

ECS Machina is this first project. It's an Entity-Component-System library. It does not hold any game logic or rendering code. It's totally agnostic, you can build anything on top of it. It's "just" a bunch of utility methods to implement efficiently the ECS pattern. The code is fully commented, documented, and has a coverage of 99%.

Entities, Components, and Systems

The ECS pattern has been gaining in popularity since few years. It forces you to decouple the logic and the data. You need to stop thinking in terms of classes and inheritance, ECS is pure composition. All objects in the game world are aggregates of Components that are picked up by Systems to be processed.

It's an great pattern for modularity, which makes it an ideal core library for any game or engine.

About the real engine

All's fine and dandy, but what about that second project, the second "smaller task"? It's a work-in-progress called Squared, and it's the real engine that I'm building on top of ECS Machina. Squared will handle the rendering, the map loading, and will ship with a bunch of systems to ease the game development as much as possible.

It still needs a bit of refactoring now that ECS Machina is done, but most of the logic is already done. The biggest work left is the documentation. and unit testing.

See you soon!

Oh and I'm quite proud of the name ECS Machina. Like, seriously.
