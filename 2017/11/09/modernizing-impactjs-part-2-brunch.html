Modernizing ImpactJS - Part 2: Time for a Brunch!

In the previous post, I explained how ImpactJS used modules before modules were a thing, and how I adapted its code for ES6. Despite the fact that ES6 modules are now natively supported by most browsers, a compilation step is still needed for a number of reasons.

  • Native ES6 modules support is pretty new, and not all browsers have this feature
  • Directly loading npm packages (and their dozens of dependencies) is awfully slow
  • Most npm packages are not even ES6-compatible
  • We'll have to compile the game for distribution anyway
  • Finally, a compilation step allows us to leverage BabelJS to use features that don't exist yet

But which JavaScript bundler/builder/compiler will we choose?

When I need to choose a tool, I tend to favor convention over configuration. It must work out-of-the-box (or at least I shouldn't spend hours on configuration). I like to spend time on my projects, not waste it to setup tools. After a bit of reviewing, I settled for Brunch. It's a bit of an underdog compared to Webpack, but it perfectly fits our needs for the ImpactJS rewrite, as it follows the KISS principle.

Ok, so what will our Brunch config file look like?

module.exports = {
  files: {
    javascripts: {
      joinTo: {
        'vendor.js': /^(?!app)/,
        'app.js': /^app\/impact/,
        'game.js': /^app\/game/,
      },
    },
    stylesheets: {joinTo: 'app.css'},
  },
  plugins: {
  babel: {
    presets: ['env'],
    plugins: [
      'syntax-dynamic-import',
      'dynamic-import-node',
    ],
  },
}

By default, Brunch watches files in the app/ folder and compile them into the public/ folder. That specific config will compile external dependencies into vendor.js, the ImpactJS engine into app.js, and our game's code into game.js. We can of course setup Brunch to make a "production" build that will bundle everything in a single file.

I've also added some Babel plugins, we'll talk about it in the next part.

Here is the new index.html that includes the compiled files, and requires the base modules:

<body>
    <canvas id="canvas"></canvas>
    <script src="./vendor.js"></script>
    <script src="./app.js"></script>
    <script src="./game.js"></script>
    <script>
        require('impact/impact')
        require('game/main')
    </script>
</body>

Since all of our code is now bundled in modules, and not automatically executed, we also need to add this "kickstart" javascript at the end of the game/main.js file:

document.addEventListener('DOMContentLoaded', () => {
  console.log('Initialized app');
  ig.main('#canvas', MyGame, 60, 320, 240, 2);
});

The original ImpactJS uses PHP as a webserver during development. Brunch now fits this role with npm start, but we can't get rid of PHP yet, because of Weltmeister.

What's that? Weltmeister is a level editor bundled with ImpactJS. It uses PHP scripts to list modules, dynamically load them, and write level files. Since we've been working on making ImpactJS ES6-friendly, Weltmeister, which is tightly integrated in ImpactJS, is basically broken. We'll work on it in the next part.
